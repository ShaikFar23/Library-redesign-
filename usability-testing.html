Part 01 — Usability Test Plan
Planning the Test Sessions
Testing Objectives
01
Evaluate how easily users can find and reserve a book using the redesigned City Library app
02
Identify usability issues and pain points in the navigation, search, and reservation flows
03
Measure task completion rates, time on task, and user satisfaction scores across 5 key tasks
04
Gather qualitative feedback on the visual design, information architecture, and content clarity
05
Provide actionable design improvement recommendations based on real user observations
06
Establish a usability baseline to measure improvements in future design iterations

Test Methodology

Moderated Think-Aloud Testing — Participants complete tasks while speaking their thoughts aloud. The facilitator observes and takes notes without guiding the user. Sessions are conducted remotely via video call and screen recording.

Participants

5 participants recruited to represent the target audience — library members aged 18–55 with varying levels of digital literacy. Mix of frequent and infrequent library users. No prior exposure to the prototype.

Tools & Equipment

Prototype: Interactive HTML prototype (CityLib app)
Recording: Screen + audio recording
Notes: Observation sheet
Survey: Post-test SUS questionnaire

Session Structure

Introduction: 5 mins — explain process, get consent
Warm-up: 3 mins — background questions
Tasks: 20 mins — 5 tasks, think aloud
Debrief: 7 mins — SUS survey + feedback

Usability Metrics to Measure
Metric Definition How to Measure Target
Task Completion Rate % of users who successfully complete each task Observe and record pass/fail per task ≥ 80%
Time on Task Time taken to complete each task Stopwatch / screen recording timestamp Under 90 sec
Error Rate Number of mistakes made per task Count wrong taps / navigation errors Under 2 errors
SUS Score System Usability Scale — overall satisfaction Post-test 10-question survey (0–100) ≥ 70 (Good)
Satisfaction Rating User's subjective rating of each task 1–5 star rating after each task ≥ 4 stars
Part 02 — Test Scenarios & Tasks
5 Tasks Given to Participants
01
Find a Book by Title
The participant is asked to find a specific book using the search feature. This tests discoverability of the search function and ease of filtering results.
"You want to borrow 'Atomic Habits' by James Clear. Find this book in the app and check if it is available."
Easy
~60 sec
02
Reserve an Available Book
The participant must navigate from search results to a book detail page and complete the reservation flow. This tests the end-to-end booking process.
"You found 'The Midnight Library' and it is available. Go ahead and reserve it for pickup at the City Centre branch."
Medium
~90 sec
03
Browse by Genre
The participant must discover and use the genre filter to browse books in a specific category. This tests the filter/navigation system discoverability.
"You're in the mood for a Sci-Fi book but don't have a specific title in mind. Browse the Sci-Fi section and find something interesting."
Medium
~75 sec
04
View Borrowed Books & Renew
The participant must navigate to their account, find their currently borrowed books, and renew one. This tests the account section and renewal flow.
"You want to check which books you currently have borrowed and renew one of them. Can you do that from within the app?"
Hard
~120 sec
05
Add a Book to Reading List
The participant must find a book that is currently checked out and add it to their reading list for later. This tests secondary actions and saving functionality.
"You found a book you like but it's checked out. Save it to your reading list so you can borrow it later."
Easy
~45 sec

Part 05 — Design Recommendations
Improvement Suggestions Based on Findings
● Critical Priority
Improve Account Navigation
4 out of 5 users struggled to find the Account section. The bottom nav icon was too small and the label "Account" wasn't associated with borrowed books.
Fix: Increase bottom nav icon size to 24px, add a "My Books" label, and add a shortcut on the home screen showing borrowed book count.
● Critical Priority
Make Filter Chips More Visible
3 users didn't realise the genre tags were tappable buttons. They appeared as plain text labels rather than interactive elements.
Fix: Add a visible border, background colour, and a subtle arrow icon to all filter chips to make them clearly interactive.
● Critical Priority
Label the Bookmark Action
3 users were unsure what the bookmark icon did. Some thought it saved the book, others thought it was a favourite button.
Fix: Replace the icon with a text button "Save to Reading List" or add a tooltip that appears on first use explaining the action.
● Major Priority
Increase Base Font Size
2 users (including the older participant) found the small text difficult to read, especially in book details and metadata.
Fix: Increase minimum body text from 9px to 12px, and ensure all text meets WCAG AA standards for font size and contrast.
● Major Priority
Add Onboarding for New Users
New users (Priya, Maya) explored randomly before finding key features. No guidance was given on first launch about main app features.
Fix: Add a 3-step onboarding overlay on first launch showing Search, Reserve, and Account features with visual callouts.
● Minor Priority
Improve Reserve Button Contrast
2 users hesitated before tapping the Reserve button, suggesting it didn't stand out enough against the page background on certain screens.
Fix: Increase button size slightly, add a shadow or border, and ensure the CTA button is always the most visually prominent element on the screen.

Overall Assessment

The CityLib app performed well with an average SUS score of 77.6 (Good). Search and reservation flows were strong. The main area for improvement is account navigation and accessibility for older users.

Next Steps

Implement the 3 critical fixes, then conduct a second round of usability testing with the same participants to measure improvement. Target a SUS score of 85+ in the next iteration.

Key Learning

Usability testing revealed issues that were not caught in the design phase — particularly around older users and icon-only navigation. Real user testing is essential for inclusive design.
